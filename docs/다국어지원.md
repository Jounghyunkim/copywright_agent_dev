# 다국어 지원(i18n) 전략

다양한 국가 법인에서 사용할 수 있도록 **서비스 결과물(생성된 국가별 카피)은 제외**하고, 프론트엔드 UI 텍스트 · 챗봇 · 스킬 언어를 다국어화하기 위한 계획 및 전략.

> 이 문서는 코드 변경 없이 수립한 계획 문서입니다. 구현은 Phase 로드맵에 따라 순차 진행.

---

## 1. 배경 및 목표

### 배경
- 현재 서비스의 UI 라벨·챗봇·스킬 프롬프트는 **한국어·영어 혼용** 상태 (한국 본사 사용 전제).
- 미국/독일/프랑스/스페인/UAE/중국/태국 등 **각 법인 마케터가 자국 언어로 서비스 이용** 필요.
- 출력물(카피)은 타겟 시장 언어로 생성되므로 다국어화 대상이 아니지만, **도구 UX는 사용자 모국어**여야 생산성이 보장됨.

### 목표
1. 사용자 모국어 UI로 학습 비용 제로화 → 법인별 도입 속도 가속
2. 챗봇·스킬이 사용자 언어로 응답 → 실무 친화성 확보
3. RTL(아랍어)·CJK(중국어)·동남아(태국어) 모두 무리 없이 렌더
4. 향후 언어 추가 시 **코드 변경 없이** 리소스만 추가하면 확장되는 구조

---

## 2. 지원 언어 & 우선순위

사용자 요청 7개 고유 언어(아랍어 중복 입력은 1개로 취합).

| 우선 순위 | 언어 | 로케일 코드 | 방향 | 비고 |
|---|---|---|---|---|
| P0 (기준) | 한국어 | `ko` | LTR | 소스 언어(현재 기반) |
| P1 | 영어 | `en` | LTR | 글로벌 공통, 다른 로케일 fallback 기준 |
| P1 | 독일어 | `de` | LTR | EU 대표 |
| P1 | 중국어(간체) | `zh-CN` | LTR | 최대 법인 |
| P2 | 프랑스어 | `fr` | LTR | EU·캐나다 |
| P2 | 스페인어 | `es` | LTR | 중남미·이베리아 |
| P2 | 아랍어 | `ar` | **RTL** | 레이아웃 미러링 필요 |
| P2 | 태국어 | `th` | LTR | 단어 공백 없음(라인 브레이크 이슈) |

**Fallback chain**: `사용자 로케일` → `en` → `ko`.

## 3. 번역 스코프 정의

### ✅ 번역 대상 (i18n 적용)
| 영역 | 예시 | 양 추정 |
|---|---|---|
| 내비게이션·버튼·라벨 | 홈, 카피라이트 검토, 저장 및 종료 | ~300 키 |
| 폼 필드 라벨·플레이스홀더·도움말 | Headline, "본문 카피", "선택 사항" | ~200 키 |
| 단계 지표·상태 메시지 | 브리핑, 분석, 전략 메시지, 완료됨 | ~80 키 |
| 알림·토스트·에러 메시지 | "저장에 실패했습니다", "서버 연결 끊김" | ~100 키 |
| 페이지 제목·부제·도움말 툴팁 | "카피 입력·리뷰 결과를 확인하고..." | ~150 키 |
| 설정·관리자 페이지 텍스트 | "관리자 설정", "사용 통계" | ~120 키 |
| 리뷰 결과 UI 라벨 | Strengths / Weaknesses / Improvements / Severity | ~40 키 |
| **챗봇(AI 어시스턴트) 시스템 프롬프트** | 사용자 언어로 응답하도록 유도 | 언어별 1~2개 프롬프트 |
| **스킬 프롬프트(SKILL.md)** | 리뷰 스킬의 시스템 프롬프트·평가 기준 | 56개 × N |
| 국가 표시 이름 | Korea, USA, Germany → 현지화 | 20개 |
| 날짜·숫자·통화 포맷 | 자동 포맷터 (Intl API) | N/A |
| **합계** | | ~1,000~1,500 키 |

### ❌ 번역 제외 (원문 유지)
- **생성된 카피 본문** — 이미 타겟 시장 언어로 생성됨 (예: 독일 타겟 카피는 독일어로 그대로 유지)
- **브랜드 용어 · LG 제품명** — `ThinQ`, `Life's Good`, `OLED`, `gram` 등 고유 브랜드 자산
- **스킬 ID** — `lg-brand-fit-check` 같은 기술 식별자
- **개발자·관리자 로그** — 콘솔 출력, DB 로그
- **문서 파일** — `/docs/*.md` (이 문서 포함)

## 4. 기술 아키텍처 전략

### 4.1 프론트엔드 i18n 프레임워크

**선택**: `react-i18next` + `i18next` + `i18next-http-backend` + `i18next-browser-languagedetector`

**선정 근거**
- React에서 사실상 표준, 안정성·커뮤니티 성숙
- **네임스페이스**로 feature별 리소스 분리 가능 (lazy loading)
- 복수형(`plural`), 보간(`interpolation`), 중첩(`nesting`) 완벽 지원
- ICU MessageFormat 호환 플러그인 존재 (복잡한 문법 필요 시)
- RTL은 별도 플러그인 없이 CSS 로직으로 해결(§4.5)

**리소스 구조**
```
frontend-v2/public/locales/
├── ko/
│   ├── common.json         # 공통 버튼/내비
│   ├── review.json         # 카피라이트 검토 페이지
│   ├── workflow.json       # 카피라이트 생성 페이지
│   ├── admin.json          # 관리자 페이지
│   └── chatbot.json        # 챗봇 관련 문구
├── en/
├── de/
├── zh-CN/
├── fr/
├── es/
├── ar/
└── th/
```

**키 네이밍 규약**
- `namespace:section.key` 형식 (예: `review:tab.copyInput`)
- 문맥을 포함 (예: `common:button.save` vs `common:button.save_and_exit`)
- 재사용 가능한 공통 항목은 `common:` 에만 정의, 중복 금지

### 4.2 Locale 감지 및 전환

**감지 우선순위**
1. 사용자가 명시적으로 선택한 로케일 (localStorage · URL `?lang=`)
2. 로그인 사용자의 프로필 `preferred_locale` (DB 저장, §4.6)
3. 브라우저 `navigator.languages`
4. 기본값: `ko`

**전환 UI**
- 사이드바 하단 "설정" 메뉴 옆에 **지구본 아이콘 + 현재 로케일 표시** (예: 🌐 KO)
- 클릭 시 드롭다운 → 8개 언어 + 국기 이모지 대체(코드 뱃지)
- 선택 시 즉시 리소스 로드 + HTML `lang`/`dir` 속성 갱신 + localStorage 저장
- DB 프로필의 `preferred_locale`에도 비동기 PATCH

### 4.3 리소스 관리·빌드

- `public/locales/` 정적 서빙 + i18next-http-backend로 런타임 로드 (lazy load)
- 빌드 시 `i18next-scanner`로 소스에서 `t('...')` 호출을 추출, 누락 키 리포트
- 각 언어 JSON은 Git에 커밋 (번역 변경 이력 추적)
- 번역 누락 시 콘솔 경고 + fallback 체인으로 무중단 렌더

### 4.4 컴포넌트 사용 패턴

**패턴 A — 단순 키**
```tsx
const { t } = useTranslation('review')
<button>{t('button.runReview')}</button>
```

**패턴 B — 보간**
```tsx
t('summary.copiesSelected', { count: selectedKeys.size })
// en: "{{count}} copies selected"
// ko: "{{count}}개 카피 선택됨"
```

**패턴 C — 복수형**
```jsonc
// en/review.json
{
  "summary": {
    "countries_one": "{{count}} country",
    "countries_other": "{{count}} countries"
  }
}
```

### 4.5 RTL(아랍어) 지원

**HTML 속성**
- `<html lang="ar" dir="rtl">` — React root에서 로케일 변경 시 자동 갱신
- `body.dir` CSS 선택자로 미러링 전용 스타일 분기

**CSS 로직**
- `margin-left` / `padding-right` → **`margin-inline-start` / `padding-inline-end`** 같은 logical properties로 전환
- 기존 스타일 파일(`globals.css` 등) 전수 스캔 → 물리 방향 속성을 논리 속성으로 치환
- 아이콘 방향성: `←`(back) 버튼은 RTL에서 `→`로 자동 전환되도록 `transform: scaleX(var(--dir-scale))` 패턴 적용 또는 아이콘 레이어에서 `dir="rtl"` 속성 이용
- 숫자·날짜는 LTR 유지 (Intl 포맷터가 locale별 처리)

**혼합 텍스트**
- 카피 결과 표시에서 "아랍어 본문 + 영문 브랜드명"이 섞이는 경우 `unicode-bidi: isolate` 필요
- 라벨은 아랍어, 카피 콘텐츠는 원문 언어 — 둘을 한 화면에 띄울 때 `dir="auto"` 속성으로 콘텐츠별 방향 자동 감지

### 4.6 폰트 전략

| 언어 | 권장 폰트 (웹폰트) | 이유 |
|---|---|---|
| ko | Noto Sans KR (현재) | 현행 유지 |
| en | Inter | 현행 유지 |
| de / fr / es | Inter (라틴 확장) | 동일 폰트로 감성 일관성 |
| zh-CN | Noto Sans SC | 간체 최적화, 라이센스 무료 |
| ar | Noto Sans Arabic | 연결 글자 완벽 처리 |
| th | Noto Sans Thai | 성조 기호 정확 렌더 |

**로딩 전략**
- 현재 `index.html`의 Google Fonts 링크에 **지연 로드**로 추가 (언어 전환 시점에만 해당 폰트 fetch)
- CSS `font-family` 스택을 언어별로 분기:
  ```css
  html[lang="ar"] { font-family: 'Noto Sans Arabic', 'Inter', sans-serif; }
  html[lang="zh-CN"] { font-family: 'Noto Sans SC', 'Inter', sans-serif; }
  html[lang="th"] { font-family: 'Noto Sans Thai', 'Inter', sans-serif; }
  ```
- 초기 페이지 무게 증가 방지 — **현재 로케일의 폰트만 로드**

### 4.7 백엔드 i18n (응답 메시지)

**대상**
- API 에러 메시지 (`"저장에 실패했습니다"`)
- SSE 진행 메시지 (`"Generated N search queries"`)
- 감사 로그 라벨

**전략**
- FastAPI에 경량 i18n 레이어 추가: 요청 헤더 `Accept-Language` 파싱 → `request.state.locale`
- 서버 번역 리소스: `backend/app/i18n/<locale>.json` (프론트와 키 스키마 일치하지 않아도 됨)
- **기계적 메시지만 서버 번역** (결과물은 아님)
- 에러 응답 포맷 확장:
  ```json
  { "code": "CORRECTION_LLM_FAIL", "message": "...", "locale": "ko" }
  ```
  프론트가 `code`로 현지화된 메시지로 다시 렌더 (fallback)

### 4.8 스킬(SKILL.md) 다국어화

**핵심 원칙**
- **스킬의 평가 대상(카피)은 원문 언어 유지**, **스킬의 시스템 프롬프트·지시문은 사용자 로케일**로 전환
- 한 번만 좋은 한국어 원문을 유지하고 영어·타겟 언어는 LLM 기반 번역 + 사람 검수로 운영

**구조 선택: 파일 분리 방식**
```
backend/skills/lg-brand-fit-check/
├── SKILL.md           # 기본(소스) = 한국어
├── SKILL.en.md        # 영어
├── SKILL.de.md        # 독일어
├── SKILL.zh-CN.md
├── SKILL.fr.md
├── SKILL.es.md
├── SKILL.ar.md
├── SKILL.th.md
└── prompt.meta.json   # 번역 상태·버전 추적
```

- `SkillLoader`에 `locale` 인자 추가 — `SKILL.<locale>.md` 없으면 `SKILL.md` (한국어) fallback
- `prompt.meta.json`이 원본 체크섬과 번역본 체크섬을 추적 → 원본 변경 시 번역 재생성 플래그
- 변경 빈도가 낮은 문서(LG 브랜드 철학 등)는 일회성 번역으로 충분

**대안: 단일 파일 섹션 방식(미채택)**
- `## System Prompt (ko)` / `## System Prompt (en)` 섹션 분리
- 장점: 파일 1개로 관리 쉬움
- 단점: 파일이 비대해지고 git blame 추적 어려움 → 7개 언어 × 56개 스킬 = 관리 지옥

### 4.9 챗봇(AI 어시스턴트) 다국어화

**사용자 언어 결정**
- 프론트가 SSE `/chat` 요청 시 현재 `i18n.language`를 `locale` 필드로 전송
- 백엔드 챗봇 시스템 프롬프트가 `locale`에 따라 "Respond in {locale_human_name}" 지시

**시스템 프롬프트 다국어화**
```python
SYSTEM_PROMPTS = {
  "ko": "당신은 LG 글로벌 마케팅 어시스턴트입니다. ...",
  "en": "You are an LG Global Marketing Assistant. ...",
  ...
}
```
- 기본 지시문만 로케일별로 정의, 카피/캠페인 데이터는 원문 그대로 주입
- 사용자 메시지에 한국어가 섞여 있어도 LLM이 현재 로케일로 응답하도록 강제

**폴백**
- 특정 로케일 프롬프트가 없으면 `en`으로 요청 후 사용자 언어로 응답을 요청하는 `Please respond in Korean/German/...` 한 줄 덧붙임

## 5. 콘텐츠(번역) 프로세스 전략

### 5.1 3-Track 병행

| Track | 용도 | 도구 | 품질 |
|---|---|---|---|
| **A. 전문 번역사** | 법적 문구·브랜드 슬로건·핵심 CTA | LSP 의뢰 | ★★★ |
| **B. LLM 번역 + 현지 법인 검수** | 일반 UI 라벨·폼 도움말·스킬 프롬프트 | GPT-4/Claude + 현지 마케터 검수 | ★★ |
| **C. LLM 번역 Only** | 챗봇 시스템 프롬프트·개발자용 진행 로그 | 자동 | ★ |

### 5.2 번역 운영 루틴

1. **소스(ko) 확정** → PR에 `i18n:needs-translation` 라벨
2. CI가 자동으로 다른 로케일 파일에 미번역 키를 플레이스홀더로 추가 (`__MISSING__`)
3. **LLM 번역 초안** 자동 생성 (빌드 파이프라인 or 수동 트리거)
4. 현지 법인 마케터가 PR에서 리뷰·수정 (GitHub Suggest 이용)
5. 머지 → 배포

### 5.3 용어집(Glossary) 구축

**반드시 특정 용어로 고정할 단어** — 법인별 일관성 확보
- 브랜드: `LG`, `ThinQ`, `Life's Good`, `OLED`, `gram` — 전 언어 원문 유지
- 기능명: "카피라이트 검토", "Copy Review", "Copy-Prüfung" → 각 언어별 표준
- Severity 라벨: Critical / Warning / Suggestion — 각 언어 공식 번역
- 단계명: 브리핑/분석/전략 메시지/카피 생성/검토

**저장 위치**
- `docs/glossary/<locale>.tsv` (탭 구분, 번역자·검수자 기록)
- 번역 파이프라인이 glossary 우선 적용

### 5.4 비주얼 QA

- 각 언어 전환 후 주요 페이지 스크린샷 비교 (Percy/Chromatic 활용 권장)
- 긴 번역(독일어는 영어 대비 1.3~1.5배)으로 **버튼 넘침** 확인
- RTL 모드 레이아웃 미러링 검증

| 1-A | `react-i18next` 도입, Provider·LanguageDetector 세팅 |
| 1-B | `public/locales/ko/`, `public/locales/en/` 구조 확립 |
| 1-C | 기존 하드코딩 문자열 → `t()` 호출로 치환 (스캐너 기반 대량 작업) |
| 1-D | 언어 전환 UI (사이드바 지구본) |
| 1-E | HTML `lang`/`dir` 자동 갱신, 폰트 스택 로케일별 분기 |
| 1-F | 사용자 프로필 DB에 `preferred_locale` 컬럼 추가 + API |
| 1-G | 영어(`en`) 번역 완료 (LLM 초안 + 내부 검수) |

**성공 기준**: 한국어/영어 스위칭 완전 동작, UI 텍스트 번역 누락 0%.

### Phase 2 — EU·CN 확장 + RTL (3주)
| 항목 | 산출물 |
|---|---|
| 2-A | 독일어·중국어(zh-CN)·프랑스어·스페인어 번역 리소스 |
| 2-B | 아랍어 RTL 레이아웃 — `margin-inline-*` 전환, 아이콘 방향 처리 |
| 2-C | Noto Sans SC·Arabic 폰트 로딩 (지연 로드) |
| 2-D | 백엔드 `Accept-Language` 파싱, 에러 메시지 다국어 JSON |
| 2-E | 챗봇 시스템 프롬프트 언어 분기 |
| 2-F | 현지 법인 Pilot — 독일·미국·중국 마케터 각 2명씩 피드백 수집 |

**성공 기준**: 6개 언어 + RTL이 기능적으로 무결, 법인 Pilot NPS 7점 이상.

### Phase 3 — 태국어 + 스킬 다국어화 (3주)
| 항목 | 산출물 |
|---|---|
| 3-A | 태국어 번역 + Noto Sans Thai 폰트 |
| 3-B | 주요 스킬(Brand / Compliance 11개) `SKILL.<locale>.md` 번역 |
| 3-C | `SkillLoader` 로케일 파라미터 지원, 자동 fallback |
| 3-D | 용어집 v1 배포 (`docs/glossary/`) |
| 3-E | 번역 품질 대시보드 — 누락 키·체크섬 불일치 집계 |

**성공 기준**: 모든 7개 언어에서 UI·챗봇·주요 스킬이 현지어 동작, 관리자 페이지에 번역 상태 가시화.

### Phase 4 — 전체 스킬 다국어화·자동화 (지속)
- 나머지 스킬(culture-*, writer-*) 번역
- 소스 변경 시 LLM 자동 번역 → PR 생성하는 CI 파이프라인
- 번역 커버리지를 관리자 페이지 대시보드로 실시간 표시

## 7. 언어별 특이사항 및 대응

### 영어 (en)
- 복수형 규칙이 명확, i18next 기본 지원
- 단어 길이 짧아 한국어 대비 공간 여유 → 레이아웃 문제 거의 없음

### 독일어 (de)
- 합성어로 단어 길이 매우 김 (예: "Rechtschreibprüfung")
- 버튼·컬럼 폭 설계 시 **+50% 여유** 권장
- 존칭(Sie) vs 친칭(du) — 비즈니스 도구는 Sie 통일

### 프랑스어 (fr)
- 성별(남/여) 형용사 변화 — 복수형 외에도 주의
- 악센트 문자(é, è, ç) — 폰트·인코딩 UTF-8 필수

### 스페인어 (es)
- 라틴아메리카 vs 이베리아 변형 차이 — 우선 중립적 표현(ES-419) 채택 권장
- 뒤집은 물음표·느낌표(¿ ¡) 렌더 확인

### 아랍어 (ar) — 가장 복잡
- **RTL 레이아웃 완전 미러링**
- 연결 글자(ligature) — 반드시 Noto Sans Arabic 같은 아랍어 전용 폰트
- 숫자는 서양 아라비아 숫자 사용 권장(Intl.NumberFormat 기본값), 동양 아라비아(١٢٣)는 옵션
- 방언 다수 — 표준 현대 아랍어(MSA)로 통일
- 아이콘 방향성 ← → 자동 뒤집힘 필요

### 중국어 간체 (zh-CN)
- 간체/번체 분리 필요 — 우선 간체만, 번체(zh-TW)는 Phase 4+
- 행간 여유 필요 (한자 가독성)
- 글자 수가 짧아져 버튼·라벨 공간 여유 생김
- 구두점: 전각(，。) 사용

### 태국어 (th)
- **단어 간 공백 없음** — 라인 브레이크 위치가 자동 결정 불가능한 경우 많음
- 성조 기호 위·아래로 적층 — 기본 폰트로는 깨짐, Noto Sans Thai 필수
- 긴 문장은 의미 단위로 `<wbr/>` 삽입 검토
- 태국 불교 달력 이슈 — 날짜는 그레고리력으로 통일 명시

## 11. 코드 변경 없이 지금 할 수 있는 준비 작업

본 전략은 "코드 변경 없음"을 전제로 수립되었으나, 다음 3가지는 **코드 변경 없이도 즉시 진행 가능**해 Phase 1 착수 속도를 올립니다.

1. **한국어 원본 문자열 추출·정리** — 현재 모든 `.tsx`·`.ts`에서 사용자 노출 문자열을 추출해 엑셀/Google Sheet에 정리. 구현 없이 먼저 번역 소스 확보.
2. **용어집 초안 작성** — 브랜드·UX·Severity 핵심 용어 50개 뽑아 법인별 초벌 번역 의뢰 (Track A 대상 사전 확정).
3. **법인별 담당자 식별** — 미국·독일·중국·프랑스·스페인·UAE·태국 법인에서 **검수 담당 마케터 1명** 선정. 이들이 Phase 1-G / 2 품질 관문.
